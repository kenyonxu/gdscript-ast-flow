架构方案对比

目标：解析 .gd 源码 → AST + 符号表 + 调用图 + 信号图

A

管道架构（推荐）

Tokenizer → Parser → SymbolResolver 三阶段独立管道。每阶段有明确的输入输出接口，可独立测试、分阶段实现

B

单体架构

单个 GDScriptAnalyzer 类内部完成所有工作。API 简单但难以扩展和测试

C

事件驱动架构

Tokenizer/Parser/Resolver 通过 GDScript 信号通信。灵活但调试困难，信号开销大

推荐 A：管道架构详情

┌─────────────────────────────────────────────────┐ │ GDScriptTokenizer │ │ Input: String (源码文本) │ │ Output: Array[Token] │ │ 职责: 词法分析，识别关键字/标识符/运算符/字面量 │ │ /缩进(INDENT/DEDENT)/换行 │ │ ~400行 │ └────────────────────┬────────────────────────────┘ │ token_list ┌────────────────────▼────────────────────────────┐ │ GDScriptParser │ │ Input: Array[Token] │ │ Output: ClassNode (AST 根节点) │ │ 职责: 递归下降语法分析，运算符优先级解析 │ │ 构建完整 AST 节点树 │ │ ~1200行 │ └────────────────────┬────────────────────────────┘ │ ast_root ┌────────────────────▼────────────────────────────┐ │ GDScriptSymbolResolver │ │ Input: ClassNode (AST) │ │ Output: AnalysisResult │ │ 职责: 遍历 AST 构建: │ │ • SymbolTable (作用域嵌套符号表) │ │ • CallGraph (方法调用关系) │ │ • SignalGraph (信号声明→连接→发射) │ │ • DefUseChain (变量定义→读写位置) │ │ ~500行 │ └─────────────────────────────────────────────────┘
